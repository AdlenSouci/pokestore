Version unique de reference

Cahier des charges - Pokemon App

Projet web + API + mobile - React, NestJS, Prisma, Stripe, Expo

Date: 16/04/2026

1) Resume executif

  • Pokemon App est une plateforme de collection et d'achat de cartes TCG avec une experience immersive.
  • Le produit est compose de trois briques: frontend, backend, mobile-rn.
  • Objectif principal: proposer un parcours complet, fluide et securise.

2) Contexte et vision

  • Positionnement gaming + e-commerce avec univers visuel fort.
  • Navigation simple, rapide et orientee conversion.
  • Coherence des usages web et mobile sur la meme API.

La vision cible couvre achat, collection et extension mobile-first.

3) Objectifs produit

  • Catalogue de cartes riche, filtre et pagine.
  • Inscription/connexion locale et Google OAuth.
  • Gestion panier complete et paiement Stripe.
  • Historique de commandes utilisateur.
  • Base technique maintainable et evolutive.

4) Cibles utilisateur

  • Collectionneur debutant: recherche par type, rarete et prix.
  • Collectionneur confirme: filtrage rapide par set/serie/annee.
  • Acheteur mobile: consultation et achat en deplacement.

5) Parcours utilisateur de reference

  1. Arrivee sur l'accueil.
  2. Entree dans la boutique.
  3. Recherche + filtres.
  4. Consultation detail carte.
  5. Ajout panier.
  6. Checkout Stripe.
  7. Suivi des commandes.

6) Perimetre fonctionnel detaille

6.1 Frontend web (frontend)

  • Accueil immersive + CTA boutique.
  • Catalogue pagine avec filtres dynamiques.
  • Detail carte enrichi et modal auth.
  • Profil, commandes, panier, transition checkout.

6.2 Backend API (backend)

  • API NestJS prefixee /api.
  • Swagger sur /api/docs.
  • Auth JWT, Stripe, Prisma/PostgreSQL.

6.3 Mobile (mobile-rn)

  • React Native / Expo connecte a la meme API.
  • Ecrans MVP: accueil, boutique, detail.
  • Alignement UX avec le frontend web.

7) Endpoints metier cibles

Auth

  • POST /api/auth/register
  • POST /api/auth/login
  • GET /api/auth/google
  • GET /api/auth/profile
  • PUT /api/auth/profile

Cards

  • GET /api/cards
  • GET /api/cards/meta
  • GET /api/cards/import

Cart

  • GET /api/cart
  • POST /api/cart
  • PATCH /api/cart/:cardId
  • DELETE /api/cart/:cardId

Orders

  • POST /api/orders/checkout-session
  • POST /api/orders
  • GET /api/orders

8) Exigences non fonctionnelles

  • Securite: JWT, guards, hash mot de passe, DTO valides.
  • Performance: pagination et reponses API structurees.
  • Accessibilite: lisibilite et contraste des ecrans critiques.
  • Qualite: architecture modules/services et typage TypeScript.

9) Contraintes techniques

  • Backend: Node.js, NestJS 11, Prisma, PostgreSQL, Stripe.
  • Frontend: React 19, TypeScript, Vite, Tailwind.
  • Mobile: React Native, Expo SDK 54.
  • Tests mobile: configuration LAN/IPv4 locale requise.

10) Livrables attendus

  • Web operationnel: auth, boutique, panier, commandes.
  • API documentee et exploitable.
  • Application mobile connectee a l'API.
  • Cahier des charges unifie MD + HTML + PDF.

11) Planning propose

  • Phase 1 (1 semaine): stabilisation API et auth mobile.
  • Phase 2 (1 semaine): UX, feedbacks UI, coherence ecrans.
  • Phase 3 (1 semaine): validation multi-supports et qualite.
  • Phase 4 (2-3 jours): livraison et corrections finales.

12) Risques et mitigation

  • Decalage auth web/mobile -> tests bout en bout sur backend commun.
  • Dependance source data -> fallback import JSON parametrable.
  • Regression visuelle -> captures de reference + verifs manuelles.
  • Reseau mobile instable -> messages d'erreur clairs.

13) KPI de succes

  • Connexion reussie > 95%.
  • Ajout panier sans erreur > 98%.
  • Affichage catalogue initial < 2 secondes.
  • Baisse continue du taux d'abandon checkout.
